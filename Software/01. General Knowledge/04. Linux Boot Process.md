# Linux Boot Process

## 1. Why Learn the Boot Process?

Before Linux applications can run, the system must go through a sequence of steps called the **boot process**.

As embedded developers, understanding this process helps us:

* Debug boards when they don’t start correctly.
* Compare how different boards (e.g., Raspberry Pi vs BeagleBone) load Linux.
* Decide where we can customize or optimize.

By the end of this lesson, you will not only **describe** the boot process but also **analyze** differences between boards.

---

## 2. The General Boot Flow

All Linux-based boards, no matter their brand, follow a similar boot sequence. But each stage has a **specific purpose**, is provided by a **different actor** (SoC vendor, board vendor, open-source project, or distribution), and produces certain **artifacts**. Let’s break it down carefully.

---

### 2.1. Power-On & Reset — BootROM

**Purpose:**
Initialize minimal hardware and decide where to load the next stage (SD/eMMC/SPI/USB/UART). BootROM is burned into the chip and *cannot* be modified.

**Who provides it:**
The **SoC vendor**. It is hard-coded in silicon.

**What it does:**

* Sets up clocks, minimal memory or SRAM.
* Reads boot configuration (boot pins/straps) to determine boot order.
* Searches for boot media, loads a small program (FSBL/SPL) into internal RAM, and jumps to it.
* Often includes recovery paths (USB or UART download mode).

**Artifacts:** None you can edit directly — it’s inside silicon. But it will try to load FSBL/MLO/MiniLoader from external storage.

**Debug tips:**

* If you see **no serial output**, suspect BootROM or power/boot pin issues.
* Some SoCs print nothing at this stage (UART not enabled yet).

---

### 2.2. First-stage bootloader (FSBL / SPL / MLO / MiniLoader)

**Purpose:**
Prepare DRAM and basic hardware so a larger bootloader can run. Without DRAM setup, the next stage cannot load.

**Who provides it:**

* Sometimes the **SoC vendor** (e.g., Rockchip MiniLoader).
* Sometimes part of **U-Boot** (SPL).
* Sometimes **board vendor** supplies it (TI’s MLO for BeagleBone).

**What it does:**

* Configure PLL, DRAM controller, pin mux.
* Perform secure boot signature checks if enabled.
* Load the second-stage bootloader (e.g., U-Boot proper) from storage into DRAM.

**Artifacts:** `SPL`, `MLO`, `miniloader.bin`. Often stored in raw boot sectors, not in a normal filesystem.

**Debug tips:**

* If DRAM init is wrong → system crashes early.
* If you see a few serial logs then hang → FSBL worked partly but couldn’t load the next stage.

---

### 2.3. Second-stage bootloader (U-Boot / Barebox / “bootloader proper”)

**Purpose:**
Provide a full-featured environment: console shell, boot scripts, filesystem access, network boot, firmware update, and finally load the kernel + DTB + initramfs.

**Who provides it:**
Usually **open-source U-Boot**, though some boards may ship with alternatives. Integrated into vendor SDKs and Linux distributions.

**What it does:**

* Initialize peripherals (MMC, USB, Ethernet).
* Provide environment variables (`bootcmd`, `bootargs`) and boot scripts (`boot.scr`, `extlinux.conf`).
* Load kernel image (`zImage`, `Image`, `uImage`), device tree blob (`.dtb`), and optional initramfs into RAM.
* Pass kernel command line (`bootargs`) and jump to kernel entry point.

**Artifacts:** `u-boot.bin`, `uImage` or `zImage`, `Image`, `*.dtb`, `initramfs.cpio.gz`, `boot.scr`.

**Debug tips:**

* If you reach a `U-Boot>` prompt → BootROM + FSBL are OK.
* Use `printenv`, `mmc rescan`, `fatls mmc 0:1` to check storage.
* Kernel boot failures often trace back to missing files, bad DTB, or wrong `bootargs`.

---

### 2.4. Linux Kernel

**Purpose:**
Initialize the OS core: decompress kernel, set up CPU, memory, subsystems, probe drivers, and mount the root filesystem.

**Who provides it:**
The **Linux distribution** or your custom build. DTB comes from vendor or is compiled from DTS source.

**What it does:**

* Decompress kernel into RAM and start execution.
* Parse **device tree (DTB)** or ACPI to discover hardware layout.
* Initialize drivers, buses, filesystems.
* Mount root filesystem (from initramfs or block device).
* If initramfs exists, run `/init` inside it, then `switch_root` into the real rootfs.

**Artifacts:** `zImage`, `Image`, `vmlinuz`, `uImage`, plus `*.dtb`.

**Important kernel cmdline example:**

```
console=ttyS0,115200 root=/dev/mmcblk0p2 rootwait rw init=/sbin/init
```

**Debug tips:**

* Kernel panics → check serial output (`dmesg`).
* Wrong DTB → devices not detected, storage not found.
* Rootfs mount failure → verify `root=` cmdline and filesystem support.

---

### 2.5. Init System (PID 1)

**Purpose:**
The first **user-space process**. It transitions the system from kernel space into a usable environment.

**Who provides it:**
Your **distribution**. Options include `systemd`, `SysV init`, or `busybox init` in embedded setups.

**What it does:**

* Mounts filesystems, starts daemons, sets up login or GUI.
* Launches services in correct order (udev, network, ssh, etc.).
* Provides the login prompt or GUI session.

**Artifacts:** `/sbin/init`, `systemd`, or init scripts in rootfs.

**Debug tips:**

* Kernel panic “No init found” → rootfs missing `/sbin/init`.
* Run `ps -p 1 -o comm=` to see what process is PID 1.

---

### 2.6 Analytical View – Why This Matters

* **Chain of trust:** Each stage may validate the next (signed images). BootROM validates FSBL, FSBL validates U-Boot, U-Boot validates kernel.
* **Where failures occur:**

  * No output at all → BootROM / power issue.
  * Stops before U-Boot prompt → FSBL problem.
  * U-Boot prompt but kernel won’t load → storage or script issue.
  * Kernel panic early → wrong DTB or missing drivers.
  * Kernel runs but no login → init/rootfs problem.
* **Developer control:**

  * BootROM: fixed, no control.
  * FSBL: minimal control.
  * U-Boot: maximum control (interactive debugging, network boot).
  * Kernel/init: you can customize source, drivers, rootfs.

---

### 2.7. Practical Debug Flow

1. **Attach serial console** — capture entire boot log.
2. If **no log** → check power rails, boot pins, SD card presence.
3. If **U-Boot available** → check environment with `printenv`, test storage/network.
4. If **kernel boots but fails** → inspect `dmesg` and `/proc/cmdline`.
5. If **init fails** → check rootfs contents, ensure `/sbin/init` exists.

### Key Takeaways for Learners

* **Same big picture**: All boards follow BootROM → FSBL → Bootloader → Kernel → Init.
* **Different actors**:

  * Raspberry Pi: Heavy use of **GPU firmware** (closed-source) instead of SPL.
  * BeagleBone: Classic **MLO + U-Boot** flow, fully open-source.
  * Luckfox: Vendor FSBL (MiniLoader), then standard **U-Boot**.
* **Where you control things**:

  * Raspberry Pi: Limited at early stage (bootcode/start.elf closed).
  * BeagleBone & Luckfox: More developer control with U-Boot and SPL.

---

## 4. Key Analysis

* **Same principle:** All boards go BootROM → Bootloader → Kernel → Init.
* **Different execution:**

  * Raspberry Pi: GPU is in charge first.
  * BeagleBone: ARM SPL + U-Boot give full developer control.
  * Luckfox: Rockchip MiniLoader bridges to U-Boot.
* **Implication for developers:**

  * Debugging Pi boot requires dealing with GPU firmware files.
  * Debugging BeagleBone or Luckfox relies mostly on U-Boot logs.

---

## 5. Activity for Learners (Apply + Analyze)

1. **Trace the flow**: Draw the boot sequence of your own board.
2. **Compare**: Identify which stage is proprietary and which is open-source.
3. **Question to analyze**: If your kernel fails to load, on which board would you have the *most control* to debug it? Why?

## 6. Conclusion

Understanding the Linux boot process helps you:

* **Predict** where problems might occur.
* **Compare** different boards logically.
* **Decide** how much control you have over customization.
