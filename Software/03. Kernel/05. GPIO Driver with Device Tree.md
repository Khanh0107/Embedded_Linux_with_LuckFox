# GPIO Driver with Device Tree

## Step 1: Device Tree Setup (Step by Step)

To make your driver work, you need to declare the GPIO pin in the Device Tree correctly. Below is a working example for the Rockchip RV1106 (Luckfox Pico board):

```dts
/***** LCD_BACKLIGHT *****/
{
    lcd_backlight: lcd-backlight {
        compatible = "gpio-bl-tdieney";
        bltd-gpios = <&gpio3 RK_PD3 GPIO_ACTIVE_HIGH>; // GPIO3_D3
        pinctrl-names = "default";
        pinctrl-0 = <&bltd_pin>;
        status = "okay";
    };
};

&pinctrl {
    lcd_backlight {
        bltd_pin: bltd-pin {
            rockchip,pins = <3 RK_PD3 RK_FUNC_GPIO &pcfg_pull_none>;
        };
    };
};

&pwm1 {
    status = "disabled";
};
```

### Explanation Line by Line

#### Node declaration

```dts
lcd_backlight: lcd-backlight {
```

* `lcd_backlight` is the **label**. You can reference it elsewhere using `&lcd_backlight`.
* `"lcd-backlight"` is the **node name**. This name is mostly cosmetic, but it should be descriptive.

---

#### Compatible string

```dts
compatible = "gpio-bl-tdieney";
```

* This tells the kernel which driver should bind to this node.
* The string must match an entry in your driver’s `of_device_id` table (`"gpio-bl-tdieney"` in your driver).
* If it doesn’t match, the probe function will never run.

---

#### GPIO property

```dts
bltd-gpios = <&gpio3 RK_PD3 GPIO_ACTIVE_HIGH>; // GPIO3_D3
```

* `bltd-gpios` is a **custom property name** that your driver looks up with `devm_gpiod_get(..., "bltd", ...)`.
* `&gpio3` means this pin belongs to GPIO controller 3.
* `RK_PD3` selects pin **Port D, Pin 3** inside GPIO3.
* `GPIO_ACTIVE_HIGH` means the backlight is **ON when the GPIO is 1**. If your circuit is inverted, use `GPIO_ACTIVE_LOW`.
* The comment `// GPIO3_D3` is optional, but helps humans remember the mapping.

⚠️ **Common mistake**: Forgetting `_gpios` or mismatching the property name. If your driver requests `"bltd"`, the property must be named `bltd-gpios`, not `backlight-gpios` or anything else.

---

#### Pinctrl binding

```dts
pinctrl-names = "default";
pinctrl-0 = <&bltd_pin>;
```

* `pinctrl-names` defines the list of states. `"default"` means this configuration is applied automatically at boot.
* `pinctrl-0` points to a group of pin settings defined later (`&bltd_pin`).

⚠️ If you omit this, the pin might still be owned by another function (like PWM), and your driver will fail with “pin already requested”.

---

#### Status property

```dts
status = "okay";
```

* Enables the node.
* If set to `"disabled"`, the driver will ignore it.

---

#### Pinctrl group definition

```dts
&pinctrl {
    lcd_backlight {
        bltd_pin: bltd-pin {
            rockchip,pins = <3 RK_PD3 RK_FUNC_GPIO &pcfg_pull_none>;
        };
    };
};
```

* This block tells the pinctrl driver:

  * Use **GPIO3 Port D Pin 3**.
  * Configure it with `RK_FUNC_GPIO` (function = GPIO, not PWM).
  * Apply pull configuration: `&pcfg_pull_none` means no pull-up or pull-down.

⚠️ If you don’t put the group inside its own subnode (`lcd_backlight { ... }`), you may get
`unable to find group for node bltd-pin`.

---

#### Disabling PWM

```dts
&pwm1 {
    status = "disabled";
}
```

* On this SoC, GPIO3_D3 is multiplexed with PWM1.
* If PWM1 is still enabled in the DTS, the kernel will block your GPIO request (`pin already requested by ff350010.pwm`).
* Setting `status = "disabled";` releases the pin so it can be used as GPIO.

---

✅ With all of these pieces in place, the kernel will successfully request and configure GPIO3_D3 as a backlight control pin, and your driver will bind to it.

## Step 2: Platform Driver Skeleton

In Linux, drivers bound to Device Tree are usually implemented as **platform drivers**.
Unlike basic character drivers that use `module_init()` and `module_exit()`, here we use **probe()** and **remove()**:

```c
static int bl_probe(struct platform_device *pdev);
static int bl_remove(struct platform_device *pdev);

static const struct of_device_id bl_of_match[] = {
    { .compatible = "gpio-bl-tdieney" },
    { }
};
MODULE_DEVICE_TABLE(of, bl_of_match);

static struct platform_driver bl_driver = {
    .probe = bl_probe,
    .remove = bl_remove,
    .driver = {
        .name = DEVICE_NAME,
        .of_match_table = bl_of_match,
    },
};
module_platform_driver(bl_driver);
```

### Explaination

#### `struct of_device_id`

<p align="center">
	<img src="../99. Images/Pasted image 20250928171816.png" >
</p>

* In practice, only the **`compatible`** field is widely used.
* Drivers create an array of these structures, for example:

```c
static const struct of_device_id bl_of_match[] = {
    { .compatible = "gpio-bl-tdieney" },
    { }
};
MODULE_DEVICE_TABLE(of, bl_of_match);
```

* When the kernel parses the Device Tree, it compares each node’s `compatible` property with this list.
* If it matches, the driver’s **`probe()`** function is called for that device.

👉 In short: `struct of_device_id` is the **bridge** between the Device Tree description and the driver.

#### `MODULE_DEVICE_TABLE(of, ...)`

- This macro exports the driver’s `compatible` strings so that the kernel’s module loader (via `udev`) can automatically load the correct driver when a matching device tree node is found.
	- **With this line** → the driver is auto-loaded when the board boots.
	- **Without it** → the driver will only work if you manually insert it using `insmod` or `modprobe`.allows auto-loading via udev (user-space device manager).

#### `struct platform_device`

This structure defines how a **platform driver** is registered with the Linux kernel.  
It provides a set of callback functions that the kernel will call when managing devices described in the **Device Tree** or platform data.

<p align="center">
	<img src="../99. Images/Pasted image 20250928125501.png" >
</p>

##### Key fields:

* **`probe()`**
  Called when a matching device (from Device Tree or board data) is found.
  → This is where you initialize hardware, request resources, and register subsystems.

* **`remove()`**
  Called when the device is removed or the driver is unloaded.
  → Used to free resources and undo what was done in `probe()`.

* **`shutdown()`**
  Called at system shutdown or reboot.
  → Ensure the hardware is left in a safe state.

* **`suspend()` / `resume()`**
  Power management callbacks for sleep/wakeup cycles.
  → Optional, but important for embedded systems and mobile devices.

* **`driver`**
  An embedded `struct device_driver`.
  → Contains metadata such as the driver’s name and the `of_match_table` for Device Tree matching.

<p align="center">
	<img src="../99. Images/Pasted image 20250928175224.png" >
</p>


* **`id_table`**
  Used when not relying on Device Tree, but instead matching by platform device IDs.

* **`prevent_deferred_probe`**
  If set, stops the kernel from retrying probe later. Normally left `false`.

#### Example

```c
static struct platform_driver bl_driver = {
    .probe = bl_probe,
    .remove = bl_remove,
    .driver = {
        .name = DEVICE_NAME,
        .of_match_table = bl_of_match,
    },
};
```

👉 In short:
`platform_driver` defines the **lifecycle** of your driver (probe → remove → suspend/resume).
The kernel calls these functions automatically when it detects a matching device.

#### `module_platform_driver()`

- Macro that expands to `module_init()` + `module_exit()`.
- Registers the driver at load time, unregisters at unload.

👉 When kernel finds a DT node with `compatible = "luckfox,gpio-backlight"`, it creates a `platform_device` and calls this driver’s **`probe()`**.

* `of_match_table` → maps the `compatible` string from the DTS.
* When the kernel finds a `"gpio-backlight"` node in the `.dtb`, it calls `bl_probe()`.
* When the device is removed or the module is unloaded, it calls `bl_remove()`.

## Step 3: Probe Function – Reading from Device Tree

The **probe function** is called when the kernel matches a device tree node (`compatible = "gpio-bl-tdieney"`) with this driver. Its role is to initialize hardware resources and register the device with the system.

Below is the complete `bl_probe()` with line-by-line explanation:

```c
static int bl_probe(struct platform_device *pdev)
{
    dev_t devt;
    struct device *dev;
    int ret;

    // 1. Get GPIO from Device Tree
    bl_gpio = devm_gpiod_get(&pdev->dev, "bltd", GPIOD_OUT_LOW);
    if (IS_ERR(bl_gpio)) {
        dev_err(&pdev->dev, "Failed to get GPIO: %ld\n", PTR_ERR(bl_gpio));
        return PTR_ERR(bl_gpio);
    }
```

* `devm_gpiod_get()` looks up the `bltd-gpios` property in the device tree.
* `"bltd"` must match the property name prefix. In the DTS we used `bltd-gpios = <...>`.
* `GPIOD_OUT_LOW` sets the pin as an output initialized to logic **0**.
* If the pin is not available or already in use, this returns an error (`-EPROBE_DEFER` or `-EINVAL`).

---

```c
    // 2. Force GPIO to be output
    ret = gpiod_direction_output(bl_gpio, 0);
    if (ret) {
        dev_err(&pdev->dev, "Failed to set GPIO as output: %d\n", ret);
        return ret;
    }

    dev_info(&pdev->dev, "GPIO direction: %d, initial value: %d\n",
             gpiod_get_direction(bl_gpio), gpiod_get_value(bl_gpio));
```

* Even though we requested it as output, we explicitly call `gpiod_direction_output()` to guarantee direction.
* This avoids bugs when the pinctrl or bootloader left the pin in an undefined state.
* Logging the direction/value helps debugging:

  * `0 = input`, `1 = output` for direction.
  * Value is whatever the pin is driving at that moment.

---

```c
    // 3. Allocate a character device number
    if (major == 0) {
        ret = alloc_chrdev_region(&devt, 0, 1, DEVICE_NAME);
        if (ret) {
            dev_err(&pdev->dev, "Failed to allocate chrdev region: %d\n", ret);
            return ret;
        }
        major = MAJOR(devt);
    }
```

* `alloc_chrdev_region()` reserves a major/minor number for your device.
* If `major == 0`, this is the first instance — so we allocate a new major dynamically.
* This ensures `/dev/lcd_backlight` has a valid device number.

---

```c
    // 4. Initialize and register cdev
    cdev_init(&bl_cdev, &fops);
    if (cdev_add(&bl_cdev, MKDEV(major, 0), 1)) {
        dev_err(&pdev->dev, "Failed to add cdev\n");
        unregister_chrdev_region(MKDEV(major, 0), 1);
        return -EINVAL;
    }
```

* `cdev_init()` binds the file operations (`.write` handler, etc.) to our device.
* `cdev_add()` registers it with the kernel so system calls (`open`, `write`) can reach our driver.
* If this fails, we must clean up the allocated device number.

---

```c
    // 5. Create device node under /dev
    backlight_class = class_create(THIS_MODULE, DEVICE_NAME);
    if (IS_ERR(backlight_class)) {
        dev_err(&pdev->dev, "Failed to create class: %ld\n", PTR_ERR(backlight_class));
        cdev_del(&bl_cdev);
        unregister_chrdev_region(MKDEV(major, 0), 1);
        return PTR_ERR(backlight_class);
    }

    dev = device_create(backlight_class, NULL, MKDEV(major, 0), NULL, DEVICE_NAME);
    if (IS_ERR(dev)) {
        dev_err(&pdev->dev, "Failed to create device: %ld\n", PTR_ERR(dev));
        class_destroy(backlight_class);
        cdev_del(&bl_cdev);
        unregister_chrdev_region(MKDEV(major, 0), 1);
        return PTR_ERR(dev);
    }
```

* `class_create()` creates a sysfs class `/sys/class/lcd_backlight/`.
* `device_create()` creates `/dev/lcd_backlight`, so user space can `echo 1 > /dev/lcd_backlight`.
* If either step fails, we clean up properly to avoid kernel resource leaks.

---

```c
    dev_info(&pdev->dev, "LCD backlight char device initialized, major: %d\n", major);
    return 0;
}
```

* Finally, we log success and return `0`.
* From now on, the device is fully operational.

## Step 4: Expose a Character Device

```c
static ssize_t bl_write(struct file *file, const char __user *buf,
                      size_t count, loff_t *ppos)
```

This function handles **user writes** to the `/dev/lcd_backlight` device.
It expects the user to write either `"1"` or `"0"` to turn the LCD backlight **ON** or **OFF**, respectively.

---

### Parameters

* **`struct file *file`**
  Represents the file object opened in userspace (`/dev/lcd_backlight`).

* **`const char __user *buf`**
  Pointer to the buffer provided by the user containing the data to write (e.g., `"1\n"`).

* **`size_t count`**
  Number of bytes the user attempts to write.

* **`loff_t *ppos`**
  File position (not used here).

---

### Implementation Details

1. **Local buffer for input**

   ```c
   char kbuf[2];
   ```

   * Holds up to **2 characters** from user input (`"0"`, `"1"`, plus possible `\0` or newline).
   * Keeps input parsing simple while preventing buffer overflow.

2. **Check for empty writes**

   ```c
   if (count == 0)
       return -EINVAL;
   ```

   * Rejects zero-length writes since no valid command can be processed.

3. **Copy data from userspace**

   ```c
   if (copy_from_user(kbuf, buf, min(count, sizeof(kbuf))))
       return -EFAULT;
   ```

   * Copies up to 2 bytes from user buffer into `kbuf`.
   * Uses `min(count, sizeof(kbuf))` to avoid overflow.
   * Returns `-EFAULT` if copying fails.

4. **Interpret user command**

   * If first character is `'1'`:

     ```c
     gpiod_set_value(bl_gpio, 1);
     pr_info("GPIO value: %d (ON)\n", gpiod_get_value(bl_gpio));
     ```

     Turns **backlight ON** and logs status.

   * If first character is `'0'`:

     ```c
     gpiod_set_value(bl_gpio, 0);
     pr_info("GPIO value: %d (OFF)\n", gpiod_get_value(bl_gpio));
     ```

     Turns **backlight OFF** and logs status.

   * Otherwise:

     ```c
     pr_info("Invalid input: %c\n", kbuf[0]);
     ```

     Logs invalid input for debugging.

5. **Return number of bytes processed**

   ```c
   return count;
   ```

   * Always return `count` to indicate all bytes were consumed.
   * Even invalid input is "consumed," but logged as error.

---

## Related File Operations

```c
static struct file_operations fops = {
    .owner  = THIS_MODULE,
    .write  = bl_write,
};
```

* Registers `bl_write()` as the handler for **write() system calls**.
* Ensures only **write operations** are supported on this character device.

## Step 5: Remove Function

Cleanup when the driver is removed:

```c
static int bl_remove(struct platform_device *pdev)
{
    device_destroy(backlight_class, MKDEV(major, 0));
    class_destroy(backlight_class);
    cdev_del(&bl_cdev);
    unregister_chrdev_region(MKDEV(major, 0), 1);
    dev_info(&pdev->dev, "Backlight driver removed\n");
    return 0;
}
```

## Recap

* **Device Tree** describes hardware (`compatible`, `gpios`).
* **Platform driver** binds via `probe()` and cleans up via `remove()`.
* **Character device** provides a user-space interface in `/dev`.
* **Key APIs**:

  * `of_get_named_gpio()` → parse GPIO from DT.
  * `devm_gpio_request_one()` → safely acquire GPIO.
  * `alloc_chrdev_region()`, `cdev_add()`, `device_create()` → create device node.

👉 This flow makes the driver **portable**: instead of hardcoding registers, you just modify the `.dts` file when moving to another board.
