# Device Tree

## 1. Motivation: Why Do We Need the Device Tree?

In the previous lessons, we wrote a **character driver** and then extended it to control **GPIO registers directly**. While this approach worked, it also exposed some **serious limitations**.

### Problem with Hardcoded Registers

When we hardcode addresses and configurations in a driver:

* **Board-specific**: The driver only works on *one exact board* with the same memory layout. If you switch to another SoC or board, all those addresses may change.
* **Not maintainable**: Imagine supporting 10 different boards—would you copy the driver 10 times and change addresses manually? That quickly becomes unmanageable.
* **Code clutter**: Driver logic gets mixed with board configuration details, making the code harder to read and maintain.

---

### The Need for Abstraction

Linux supports **hundreds of boards and SoCs**. Each one has:

* Different memory maps
* Different interrupt numbers
* Different peripherals enabled

The **Linux kernel developers** realized they needed a way to:

1. Keep **drivers generic and reusable**.
2. Move **board-specific details out of the driver code**.
3. Allow the **same driver to work across different hardware platforms** with only configuration changes.

---

### The Solution: Device Tree (DT)

The Device Tree provides a **hardware description language** for the kernel. Instead of the driver “knowing” the hardware details, we describe them in a **separate text file** (the Device Tree Source, or **DTS**).

* The **bootloader** passes this hardware description (compiled into a DTB file) to the kernel at startup.
* The **kernel reads the Device Tree**, learns about the devices, their addresses, interrupts, GPIO pins, etc.
* The **driver then queries the Device Tree** instead of hardcoding values.

This means:

* A single driver can run on many boards.
* If you change the hardware, you just update the Device Tree, not the driver code.
* It creates a **clean separation** between hardware description (board-level) and driver logic (kernel-level).

👉 In short: **Device Tree makes Linux portable and scalable across many boards without rewriting drivers.**

## 2. Device Tree Basics

### 2.1 The Key File Types

1. **`.dts` (Device Tree Source)**

   * Board-specific file.
   * Example: `rv1106-luckfox-pico-ultra-w.dts`.
   * Defines memory size, GPIO mappings, LEDs, buttons, etc.

2. **`.dtsi` (Device Tree Include)**

   * Shared definitions, like “headers” in C.
   * Example: `rv1106.dtsi` defines CPU, clocks, buses.

3. **`.dtb` (Device Tree Blob)**

   * Binary compiled from `.dts`.
   * Loaded into RAM by U-Boot, passed to the kernel.

4. **`.dtc` (Device Tree Compiler)**

   * Tool to compile sources into binary.
   * Example:

     ```bash
     dtc -I dts -O dtb -o rv1106-luckfox-pico-ultra-w.dtb rv1106-luckfox-pico-ultra-w.dts
     ```

### 2.2 Syntax and Conventions

#### Nodes and Labels

```dts
/ {
    // Create a new node syntax
    gpio0: gpio@ff400000 {
        compatible = "rockchip,gpio-bank";
        reg = <0xff400000 0x100>;
    };
};
```

* `gpio0` → label (used for references).
* `gpio@ff400000` → node name (gpio) with base address (ff400000).

#### Properties

In the Device Tree, **properties** are key-value pairs that describe hardware details for each node.  
They tell the Linux kernel and drivers **how to configure the hardware**.
- Some properties are **universal** (almost every device has them).
- Others are **device-specific** and must be checked in the kernel’s binding documentation.

Example:

```dts
i2c@ff150000 {
    compatible = "rockchip,rv1106-i2c";
    reg = <0xff150000 0x1000>;
    interrupts = <GIC_SPI 32 IRQ_TYPE_LEVEL_HIGH>;
    clocks = <&cru CLK_I2C1>;
    status = "okay";
};
```

* `compatible` → let the kernel know which driver to bind.
* `reg` → the base address and size of the device's register space in the memory map.
* `interrupts` → declare the interrupt that the device uses.
* `clocks` → reference the clock source (via phandle).
* `status` → `"okay"` means the device is enabled, `"disabled"` means it is not used.

⭐ Note:
- Each peripheral (e.g., I²C, SPI, camera, display, codec) defines its own additional properties.
- Examples:
	* An I²C controller might have `i2c-scl-rising-time-ns`.
	* A camera sensor might have `rotation` or `orientation`.
	* A display might use `bus-format` or `pixelclock`.

- You **don’t memorize these**. Instead:
	
	* Look in the kernel source:
	
	  ```
	  Documentation/devicetree/bindings/<subsystem>/<device>.yaml
	  ```
	* Example on Luckfox: the RV1106 I²C binding is defined in `Documentation/devicetree/bindings/i2c/rockchip,i2c.yaml`.

#### Modify a Node

If you want to change a property in above gpio0 node:

```dts
&gpio0 {
    compatible = "rockchip,gpio-bank,tdieney";
};
```

or

```dts
/ {
    gpio0: gpio@ff400000 {
        compatible = "rockchip,gpio-bank";
        reg = <0xff400000 0x100>;
    };

    gpio@ff400000 {
        compatible = "rockchip,gpio-bank,tdieney";
    };
};
```

- ⭐ Note: 
	- Preferred way: use `&gpio0` override. Redeclaration can cause duplication unless used carefully.
	- Please avoid to modify .dtsi files which provide SoC-level definitions. Instead, override or extend them in your board .dts file.

#### Phandles (References)

In Device Tree, **phandles** is a "pointer" or "handle" that allows one node to reference another node.

```dts
leds {
    compatible = "gpio-leds";

    status_led: led-0 {
        gpios = <&gpio0 10 GPIO_ACTIVE_HIGH>;
        label = "green:status";
    };
};
```

* `status_led:` is **label** for the `led-0` node.
* In another node, you can reference it using `&status_led` instead of writing the full path.
* This helps **reuse nodes** and reduce duplication.

👉 Practical: In the driver, the kernel will rely on phandles to know "which GPIO this LED is connected to", or "where this device gets its clock/reset from".

#### Includes

Device Tree is divided into multiple files for modularity:

* `.dtsi` (include file) contains common hardware definitions for a SoC or board.
* `.dts` (board file) contains specific definitions for a particular board (which GPIOs, peripherals are enabled, etc.).

Example:

```dts
#include "rv1106.dtsi"
```

* `rv1106.dtsi` defines the common SoC-level nodes (CPU, clocks, buses).
* File `.dts` for Luckfox Pico Ultra W includes this `.dtsi` and adds board-specific nodes (LEDs, buttons, memory size).

👉 That means: `.dtsi` = "blueprint of the house", `.dts` = "detailed design for a specific apartment".

### 2.3 Device Tree Hierarchy

* **Root node `/`** → model, compatible, memory.
* **SoC node** → CPU, clocks, buses.
* **Peripheral nodes** → GPIO, UART, I2C, etc.
* **Board-specific nodes** → LEDs, buttons, sensors.

Example from **Luckfox Pico Ultra W**:

```dts
/ {
    model = "Luckfox Pico Ultra W";
    compatible = "luckfox,pico-ultra-w", "rockchip,rv1106";

    memory@40000000 {
        device_type = "memory";
        reg = <0x40000000 0x10000000>; /* 256 MB */
    };
};
```

### 2.4 From Source to Kernel

1. Board `.dts` includes SoC `.dtsi`.
2. Compiled via `dtc` → `.dtb`.
3. U-Boot loads **kernel + .dtb** into memory.
4. Kernel parses `.dtb` and initializes hardware.

### 2.5 Runtime Location of Device Tree

* After boot, `.dtb` is in RAM.
* Kernel exposes it via:

  ```
  /sys/firmware/devicetree/base/
  ```
  
* Each property appears as a file.
* Example: `cat /sys/firmware/devicetree/base/model` → `Luckfox Pico Ultra W`.
* This runtime tree is read-only and reflects the active DTB, useful for debugging and confirming which DTS was used at boot.

### 2.6 Practical Debugging Tips

* If driver doesn’t load → check if `status = "okay"`.
* If hardware not detected → confirm `.dtb` is correct (`/proc/device-tree/model`).
* Old DTB still loading? → Update U-Boot configs (`extlinux.conf` or boot script).
* Always inspect `/sys/firmware/devicetree/` for live view of active nodes.

## 3. How the Kernel Uses Device Tree

* **Parsing at boot**: Bootloader loads kernel + DTB.
* **Kernel tree walk**: `of_*` API to find nodes.
* **Driver binding process**:

  * Match `compatible`.
  * Get resources (reg, irq, gpios).
* **Key APIs**:

  * `of_get_property()`
  * `of_find_node_by_name()`
  * `platform_get_resource()`
  * `of_get_named_gpio()`

---

## 4. Board Files vs Device Tree: An Analytical View

Earlier kernels used **board files**, where hardware details were hardcoded in C. This made sense when one kernel only supported one board.

But as SoCs and boards multiplied, this approach became unscalable. Each new board meant touching kernel code, recompiling, and risking regressions.

**Device Tree (DT)** solves this by moving hardware description into external `.dts` files. The kernel becomes reusable across boards; only the DT changes.

---

### Quick Comparison

| Aspect      | Board Files (Old)           | Device Tree (Modern)     |
| ----------- | --------------------------- | ------------------------ |
| Flexibility | Low – code changes required | High – edit `.dts` only  |
| Reusability | One kernel ↔ one board      | One kernel ↔ many boards |
| Debugging   | Recompile kernel each test  | Swap `.dtb` and reboot   |
