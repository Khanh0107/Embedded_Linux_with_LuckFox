# Character Drivers

## 0. Introduction

### Why Learn Character Drivers (Especially in Embedded)?

* **Core of the Linux Driver Model**
  Most peripherals (UART, I2C, SPI, GPIO, sensors, custom hardware) are exposed as **character devices**.
  Understanding character drivers gives you the foundation for other driver types.

* **Bridge Between User-Space and Kernel-Space**

  * From user-space: you call familiar APIs like `open()`, `read()`, `write()`.
  * From kernel-space: the driver implements these functions in `struct file_operations`.
    Learning this means you truly understand how **data flows from an application down into hardware**.

* **Stepping Stone for Advanced Topics**
  Once you master character drivers, learning block, network, platform drivers, or device tree bindings becomes easier.

---

## 1. Theoretical Foundations

### 1.1 Linux Device Model: *Everything is a File*

* In Linux, almost all resources (files, sockets, hardware devices) are abstracted as **files**.
* Hardware appears in `/dev` as device files.
* From user-space, interacting with hardware feels like file I/O:

```c
open("/dev/ttyS0", O_RDWR);
read(fd, buf, len);
write(fd, buf, len);
close(fd);
```

👉 The application doesn’t need to know driver details — only the device file path.

---

### 1.2 Character vs. Block Drivers

* **Character Driver**

  * Byte stream, sequential access.
  * Examples: UART, I2C, SPI, GPIO, sensors.

* **Block Driver**

  * Access data in fixed-size blocks, supports random seek.
  * Examples: HDD, SSD, SD cards.

👉 In Embedded systems, **most custom drivers (LEDs, sensors, buses)** are character drivers.

---

### 1.3 Major/Minor Numbers

* Each device has a `(major, minor)` pair:

  * **Major** = which driver handles the device.
  * **Minor** = which device instance (if multiple).

Example:

```bash
ls -l /dev/ttyS0
crw-rw---- 1 root dialout 4, 64
# "c" = char device, Major=4, Minor=64
```

---

### 1.4 System Call Flow: User → Kernel → Hardware

```
+------------------+       +-------------+       +------------------+       +----------------+
| User Application |       |   Syscall   |       | Virtual File Sys |       |  Character     |
|  echo "Hi" > dev |-----> | (open/read/ |-----> | (lookup major/   |-----> |  Driver (fops) |
|  cat /dev/hello  |       | write/...)  |       | minor, dispatch) |       |                |
+------------------+       +-------------+       +------------------+       +----------------+
                                                                               |
                                                                               v
                                                                         +-----------+
                                                                         | Hardware  |
                                                                         +-----------+
```

## 2. Core APIs in Character Drivers

### 2.1 `struct file_operations`

Defines how syscalls map to driver functions:

```c
struct file_operations {
    .owner   = THIS_MODULE,
    .open    = my_open,
    .read    = my_read,
    .write   = my_write,
    .release = my_release,
};
```

* `open()` → called on `open("/dev/mydev")`
* `read()` → called on `cat /dev/mydev`
* `write()` → called on `echo "X" > /dev/mydev`
* `release()` → called on `close(fd)`

👉 Implement only what you need.

---

### 2.2 Registering a Character Device

### 2.2 `cdev` and Character Device Registration

To make the kernel aware of your character driver, you must:

1. Allocate a **device number** (major/minor).
2. Initialize and register a `cdev` structure.
3. Clean up properly on module exit.

---

#### a. Allocate Device Numbers

**Declaration:**

```c
int alloc_chrdev_region(dev_t *dev, unsigned baseminor,
                        unsigned count, const char *name);
```

* **`dev`**: output parameter, will hold the first allocated device number.
* **`baseminor`**: starting minor number (usually `0`).
* **`count`**: number of consecutive minor numbers to allocate.
* **`name`**: device name shown in `/proc/devices`.

**Example:**

```c
alloc_chrdev_region(&dev_num, 0, 1, "mychardev");
```

👉 Allocates one device (`count = 1`) with an automatically assigned **major number** and minor = 0.

---

#### b. Initialize and Add `cdev`

**Declarations:**

```c
void cdev_init(struct cdev *cdev, const struct file_operations *fops);
int  cdev_add(struct cdev *cdev, dev_t dev, unsigned count);
```

* **`cdev`**: structure representing your character device.
* **`fops`**: pointer to your driver’s `struct file_operations`.
* **`dev`**: device number allocated earlier.
* **`count`**: number of devices this `cdev` controls.

**Example:**

```c
cdev_init(&my_cdev, &fops);     // Link fops to the cdev
cdev_add(&my_cdev, dev_num, 1); // Register with kernel
```

👉 Now the kernel knows that when user space opens `/dev/` with this major/minor, it should call your driver’s `file_operations`.

---

#### c. Cleanup on Exit

**Declarations:**

```c
void cdev_del(struct cdev *cdev);
void unregister_chrdev_region(dev_t dev, unsigned count);
```

* **`cdev_del`**: removes the character device from the kernel.
* **`unregister_chrdev_region`**: releases the allocated major/minor numbers.

**Example:**

```c
cdev_del(&my_cdev);
unregister_chrdev_region(dev_num, 1);
```

👉 Always undo what you registered in `init()` to avoid resource leaks.

**Flow Diagram:**

```
insmod hello_char.ko
        │
        ▼
+----------------------------+
| hello_init()               |
|                            |
| alloc_chrdev_region(...)   | → Reserve major/minor
| cdev_init(&my_cdev, &fops) | → Bind syscalls to driver
| cdev_add(&my_cdev, ...)    | → Register with kernel
+----------------------------+
        │
        ▼
/dev/hello_char available
```

### 2.3 Creating a Class and Device Node in `/dev`

After registering a `cdev`, the driver is known to the kernel, but user space still has no way to access it unless a device node exists in `/dev`.
This is where **`class_create()`** and **`device_create()`** come into play.

---

#### a. Create a Device Class

**Declaration:**

```c
struct class *class_create(struct module *owner, const char *name);
```

* **`owner`**: usually `THIS_MODULE`.
* **`name`**: the class name, will appear under `/sys/class/<name>/`.

**Example:**

```c
hello_class = class_create(THIS_MODULE, "hello_char");
if (IS_ERR(hello_class)) {
    pr_err("Failed to create class\n");
    return PTR_ERR(hello_class);
}
```

👉 This creates a class entry in sysfs (`/sys/class/hello_char`), grouping devices of the same type.

---

#### b. Create a Device File Automatically

**Declaration:**

```c
struct device *device_create(struct class *class, struct device *parent,
                             dev_t devt, void *drvdata,
                             const char *fmt, ...);
```

* **`class`**: pointer to the class created with `class_create()`.
* **`parent`**: parent device (can be `NULL` if not part of a bus).
* **`devt`**: device number (major/minor) allocated earlier.
* **`drvdata`**: optional driver-specific data (often `NULL`).
* **`fmt`**: format string for the device name (e.g., `"hello_char"`).

**Example:**

```c
device_create(hello_class, NULL, dev_num, NULL, "hello_char");
```

👉 This will automatically create `/dev/hello_char`, so user applications can simply `open("/dev/hello_char")`.

---

#### c. Cleanup on Exit

**Declarations:**

```c
void device_destroy(struct class *class, dev_t devt);
void class_destroy(struct class *class);
```

* **`device_destroy`**: removes the device node in `/dev`.
* **`class_destroy`**: deletes the class entry in `/sys/class/`.

**Example:**

```c
device_destroy(hello_class, dev_num);
class_destroy(hello_class);
```

👉 Always pair these with `device_create()` and `class_create()` in `exit()`.

---

⚠️ **Note**: Without a `class`, you cannot use `device_create()`.
If you only register `cdev`, the kernel knows about your driver, but **user space won’t see any `/dev/xxx` node** unless you either:

* manually create it with `mknod`, or
* let the driver auto-create it via `class_create()` + `device_create()` (preferred).

## 3. Implementation (Demo Code)

The demo driver (`hello_char`) is a simple Linux character device driver.

```c
#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/init.h>
#include <linux/fs.h>
#include <linux/cdev.h>
#include <linux/uaccess.h>

#define DEVICE_NAME "hello_char"

static dev_t dev_num;
static struct cdev hello_cdev;
static struct class *hello_class;

static char kernel_buf[128] = "Hello from kernel!\n";

static int hello_open(struct inode *inode, struct file *file) {
    pr_info("hello_char: device opened\n");
    return 0;
}

static int hello_release(struct inode *inode, struct file *file) {
    pr_info("hello_char: device closed\n");
    return 0;
}

static ssize_t hello_read(struct file *file, char __user *buf, size_t len, loff_t *offset) {
    return simple_read_from_buffer(buf, len, offset, kernel_buf, strlen(kernel_buf));
}

static ssize_t hello_write(struct file *file, const char __user *buf, size_t len, loff_t *offset) {
    size_t to_copy = min(len, sizeof(kernel_buf) - 1);
    if (copy_from_user(kernel_buf, buf, to_copy))
        return -EFAULT;
    kernel_buf[to_copy] = '\0';
    pr_info("hello_char: received from user: %s\n", kernel_buf);
    return to_copy;
}

static struct file_operations fops = {
    .owner   = THIS_MODULE,
    .open    = hello_open,
    .release = hello_release,
    .read    = hello_read,
    .write   = hello_write,
};

static int __init hello_init(void) {
    if (alloc_chrdev_region(&dev_num, 0, 1, DEVICE_NAME) < 0) {
        pr_err("Failed to allocate character device region.");
        return -1;
    }
    cdev_init(&hello_cdev, &fops);
    cdev_add(&hello_cdev, dev_num, 1);

    // Create class and device file automatically
    hello_class = class_create(THIS_MODULE, DEVICE_NAME);
    if (IS_ERR(hello_class)) {
        pr_err("Failed to create class."); 
        cdev_del(&hello_cdev);
        unregister_chrdev_region(dev_num, 1);
        pr_info("hello_char: unregistered\n");
        return -1; 
    }
    return 0;
}

static void __exit hello_exit(void)
{
    device_destroy(hello_class, dev_num);   /* delete /dev/hello_char */
    class_destroy(hello_class);             /* delete /sys/class/hello_char */
    cdev_del(&hello_cdev);
    unregister_chrdev_region(dev_num, 1);
    pr_info("hello_char: unregistered\n");
}

module_init(hello_init);
module_exit(hello_exit);

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Tdieney");
MODULE_DESCRIPTION("Simple Driver");
MODULE_VERSION("1.0");

```

Its main functions:

* **Initialization (`hello_init`)**

  * Allocate device number (`alloc_chrdev_region`)
  * Initialize and register `cdev`
  * Create device class and `/dev/hello_char` automatically

* **Exit (`hello_exit`)**

  * Remove device node
  * Destroy class and cdev
  * Unregister device number

* **File Operations (`struct file_operations`)**

  * `open()` / `release()` → log when device is opened/closed
  * `read()` → copy data from kernel buffer to user space
  * `write()` → copy data from user space into kernel buffer

## 4. Analysis of Driver Behavior

When interacting with `/dev/hello_char`, the flow between **user space** and **kernel space** is as follows:

```
   +------------------+                      +------------------+
   |   User Process   |                      |   Kernel Driver  |
   +------------------+                      +------------------+

   echo "Hi" > /dev/hello_char
   ──────────────────────────────▶  hello_write()
                                    ┌──────────────────────┐
                                    │ copy_from_user()     │
                                    │ → kernel_buf[]       │
                                    │ pr_info("Hi")        │
                                    └──────────────────────┘

   cat /dev/hello_char
   ◀───────────────────────────────  hello_read()
                                    ┌──────────────────────┐
                                    │ simple_read_from_buf │
                                    │ kernel_buf[] → user  │
                                    └──────────────────────┘
```

* **Write path (`echo`)**: user data is transferred to `kernel_buf` using `copy_from_user`.
* **Read path (`cat`)**: kernel buffer content is copied back to user space using `simple_read_from_buffer`.
* All interactions are logged via `pr_info` → can be verified with `dmesg`.

👉 This driver demonstrates the fundamental mechanism of a character driver:

* Communication via **file operations**
* Data transfer using **user ↔ kernel buffer**
* Device node `/dev/hello_char` as the interface

---

## 5. Summary

* Everything is a file in Linux.
* Character drivers map syscalls to kernel functions.
* Major/minor numbers link `/dev` files to drivers.
* Memory safety (`copy_to_user`, `copy_from_user`) is critical.
* Concurrency must be handled explicitly.
